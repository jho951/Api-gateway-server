# 중앙 SSO와 하이브리드 JWT 흐름

## 1. 목적
- 중앙 SSO와 하이브리드 JWT 토큰을 함께 활용해 인증/인가를 단일화하면서도 API 게이트웨이 단에서 유연하게 권한 검증 및 세션 관리를 수행한다.

## 2. 아키텍처 
- **사용자/클라이언트**: SSO 로그인 후 인증 정보를 가진 상태로 API 요청을 보낸다.
- **SSO IDP(IdP)**: 인증 처리 후 JWT 및 사용자 메타정보를 발급한다.
- **API 게이트웨이**: 중앙 SSO 토큰을 검증하고, 필요하면 자체 서명된 JWT로 교환하거나 세션 쿠키를 관리한다.
- **마이크로서비스**: 게이트웨이가 전달한 JWT/세션 정보를 기반으로 모든 권한 검증을 수행한다.

## 3. 중앙 SSO 흐름
1. 사용자 → SSO 인증 페이지 요청
2. IDP에서 자격 증명 검증
3. 인증 토큰(ID 토큰 및 액세스 토큰)과 함께 리디렉션
4. 브라우저/클라이언트가 API 게이트웨이로 요청할 때 Authorization 헤더에 토큰 포함
5. 게이트웨이가 토큰 서명, 만료, 조작 여부를 검증
6. 검증이 성공하면 사용자 정보(권한, user_id 등)를 추출하여 이후 흐름으로 전달

## 4. 하이브리드 JWT 흐름
### 4.1 목적
- SSO 토큰이 만료되거나 외부 파편화된 서비스에서 자체 검증을 요구할 경우 게이트웨이에서 자체 JWT를 발급해 재사용 시간을 연장하거나 필요한 청구 정보를 포함.

### 4.2 단계
1. 게이트웨이가 **중앙 SSO 토큰**을 검증 후, 자체 **하이브리드 JWT**를 생성.
   - 페이로드에 `sub`, `exp`, `scope`, `orig_token_hash`, `client_id`, `issued_at` 등을 포함.
2. 게이트웨이는 하이브리드 JWT를 API 요청 헤더 또는 쿠키로 응답.
3. 클라이언트는 이후 요청에서 하이브리드 JWT를 Authorization 헤더에 사용.
4. 마이크로서비스는 게이트웨이의 키세트(공개 키)로 직접 하이브리드 JWT를 검증.
5. 게이트웨이는 필요 시 토큰 재발급(리프레시) 또는 SSO 세션 연장을 위해 IDP와 통신.

### 4.3 고려 사항
- **토큰 수명**: SSO 토큰은 짧게, 하이브리드 JWT는 게이트웨이 보유 정책에 따라 유연하게 설정.
- **토큰 연동**: `orig_token_hash`를 넣어 하이브리드 JWT가 어떤 SSO 세션과 연결됐는지 추적.
- **리프레시 전략**: 하이브리드 JWT 만료 시 게이트웨이에서 SSO IDP에 `refresh_token` 요청 후 다시 발급.

## 5. 실패 처리 및 롤백
- SSO 토큰 검증 실패 → 401 응답, 클라이언트에게 로그인 화면 리디렉션
- 하이브리드 JWT 검증 실패 → 게이트웨이에서 재발급 시도, 재시도 불가 시 401

## 6. 장점 요약
- 중앙 인증 유지: IDP에서 누구나 동일한 정책으로 로그인 가능
- 게이트웨이의 빠른 인가: 하이브리드 JWT를 통해 마이크로서비스마다 SSO 검증 없이 권한 검사
- 유연한 세션 연장: 자체 토큰이 있기 때문에 필요한 경우만 IDP에 요청
